1. EXECUTIVE SUMMARY

* Project Overview
FaceShield EdgeAI is a highly accurate, lightweight, and entirely offline facial recognition, personal protective equipment (PPE) compliance, and passive liveness detection solution. Developed specifically for extreme environments and high-security zones, FaceShield EdgeAI is designed to be integrated seamlessly into the existing National Highways Authority of India (NHAI) Datalake 3.0 ecosystem. By combining local edge-intelligence models with a multi-signal identity trust framework, FaceShield EdgeAI delivers zero-trust biometric authentication, incident monitoring, and automatic data reconciliation without relying on external cloud biometrics or active internet connections.

* Vision
Our vision is to secure NHAI's vast infrastructure network by establishing an uncompromisable, zero-network, zero-fraud identity frontier. FaceShield EdgeAI guarantees that every individual entering an NHAI construction site, toll plaza, or asset yard is uniquely verified, fully compliant with safety regulations, and authenticated instantly—even in the deepest tunnels or high-altitude mountain passes where networks do not exist.

* Core Innovation
At the heart of FaceShield EdgeAI are four core technical innovations discovered directly within our codebase:
1. Local Multi-Signal Passive Liveness Engine: A three-tiered offline validation pipeline combining Laplacian Texture Variance, Specular Highlight Analysis, and HSV Skin-Tone Saturation distribution to block static photos, digital screen replays, and mask attacks without expensive deep-learning models or active connections.
2. Composite Identity Trust scoring: Replacing binary "Match/Mismatch" verification with a multi-factor confidence index (GPS distance, hardware trust, historical behavior, face similarity, and liveness scores) to enforce high-assurance access decisions.
3. Decoupled Dual-Database Caching: Mirroring the PostgreSQL relational database (with high-performance pgvector indexes) onto a local, thread-safe SQLite cache (fencein_cache.db) for instantaneous offline 1:N searches.
4. Edge-Optimized Computer Vision: Real-time OpenCV-based PPE detection (helmets, high-vis vests, face masks) running locally on CPU-only edge hardware alongside UltraFace and ArcFace ONNX neural models.

* Alignment with NHAI Challenge
The NHAI Innovation Hackathon 7.0 Challenge Statement demands a highly accurate, lightweight, and entirely offline biometric and liveness detection algorithm that integrates seamlessly into the existing Datalake 3.0 application. FaceShield EdgeAI satisfies this by executing all neural inferences, database comparisons, and safety audits at the physical edge gate. All transaction logs and incident feeds are packaged in the NHAI Unified Event Format (UEF-1.0) and stored in a local SQLite event queue. The moment connectivity is restored, a custom Service Worker background sync reconciles the queue; in permanently airgapped zones, the engine supports secure physical exports for manual Datalake ingestion.

2. PROBLEM STATEMENT

* Existing Challenges in Identity Verification
Identity verification at highway development projects and remote infrastructure assets remains highly fragmented. Traditional security relies on RFID badges or visual inspections by guards. These systems suffer from high rates of "buddy punching," contractor fraud, and unauthorized badge swapping, resulting in massive leakage in payroll budgets, security vulnerabilities, and compliance liabilities.

* Issues in Remote and Zero-Network Zones
NHAI operates across highly challenging geographical terrains, including mountainous regions, dense forests, and underground tunnels. In these zero-network zones, cloud-based biometric systems fail completely, defaulting to manual overrides that bypass security controls. Existing commercial facial recognition systems require continuous internet access to query central databases or execute expensive cloud API models, making them useless for continuous, high-speed operations at the physical edge.

* Fraud Risks
Contractor fraud manifests in two primary vectors:
1. "Ghost Workers": Relational and identity duplication where the same individual is enrolled under multiple contractor rosters to draw double pay.
2. Biometric Spoofing: Bad actors presenting printed high-resolution photographs, pre-recorded videos on mobile screens, or realistic masks to trick standard camera gates.

* Operational Bottlenecks
Centralized biometric gateways introduce severe latency, especially when dealing with hundreds of shifts starting simultaneously. Network jitter, packet loss, and high round-trip-time (RTT) to centralized cloud servers cause gate congestion, leading supervisors to manually bypass biometrics to avoid shift delays.

* Need for Offline Authentication
There is an operational requirement for a self-contained, offline-first edge biometric gateway. This gateway must operate with absolute autonomy, performing 1:N facial matching against thousands of local profiles, auditing safety equipment compliance, logging incidents, and queuing audit records locally, all while requiring zero active network packets.

3. PROPOSED SOLUTION

* Detailed Solution Overview
FaceShield EdgeAI is an enterprise-grade, edge-processed command and control biometrics system. It operates via a localized microservice architecture running a Python FastAPI biometric service alongside a React 19 Progressive Web App (PWA) client and a NestJS API gateway. The entire verification pipeline is optimized to run locally on low-power CPU edge nodes (such as Raspberry Pi 4/5, smart kiosks, or industrial tablets) with zero cloud dependency.

* Offline Architecture
The system establishes complete network independence through a decoupled dual-database mirror strategy. The primary cloud PostgreSQL database leverages the pgvector extension to store 512-dimensional face embeddings. The edge terminal runs a local thread-safe SQLite database (offline_cache.py) which acts as an active read-only mirror of the PostgreSQL database within the tenant context. A localized synchronization loop (sync_embeddings_from_pg) populates the SQLite mirror at startup. During offline operations, 1:N cosine similarity searches are executed in memory using NumPy against the SQLite cache, completing matches in milliseconds with zero network overhead.

* Facial Recognition Engine
The facial recognition engine is built using CPU-optimized ONNX Runtime neural engine sessions:
1. Face Detection: UltraFace (version-RFB-320.onnx) runs on the raw camera frame (resized to 320x240) to detect faces with high robustness. Non-Maximum Suppression (NMS) with an IoU threshold of 0.3 filters overlapping bounding boxes, selecting the primary face above a 0.70 confidence threshold.
2. Embedding Extraction: The cropped face is normalized and passed through the ArcFace (arcface.onnx) ONNX model to generate deep, L2-normalized 512-dimensional vector embeddings. These vectors represent unique face keypoints, enabling deterministic similarity calculations.

* Liveness Detection Engine
Rather than relying on resource-intensive deep learning networks, FaceShield EdgeAI uses a high-speed, 3-Signal Passive Liveness Detection Engine designed to run on low-power CPUs:
1. Laplacian Texture Variance: Live human skin exhibits high-frequency details. The engine computes the Laplacian variance of the face crop; flat screens or printed paper appear blurred and score extremely low.
2. Specular Highlight Analysis: Real human skin produces bright glare spots at specular angles under ambient light. The engine tracks bright pixels (>230 grayscale value) to expect a 0.5% to 8% specular coverage, penalizing matte printouts.
3. HSV Skin-Tone Saturation Distribution: A live face possesses a wide distribution of skin hues. The engine isolates skin-tone hue bands in HSV space (Hue [0,25] U [160,180]) and computes the standard deviation of saturation, detecting grayscale printouts or compressed IR replays.
A weighted composite liveness score >= 0.40 confirms a live subject.

* Edge AI Capabilities
Beyond identity verification, the edge engine executes real-time safety auditing via an OpenCV-based Personal Protective Equipment (PPE) Compliance Detector:
1. Helmet Detection: Analyzes the top 35% of the frame (head region) in HSV space to match common hard hat colors (yellow, orange, white, red, blue). If >=3% of the region matches, helmet compliance is confirmed.
2. Safety Vest Detection: Crops the middle 35%-80% of the image (torso region) to detect bright reflective orange, yellow, and lime-green vest pigments (requires >=5% coverage).
3. Face Mask Detection: Examines the lower 45% of the face bounding box. If skin ratio is <12%, a protective mask is detected.
Access is dynamically blocked if required PPE items are missing.

* Integration Strategy with Datalake 3.0
The adapter (datalake_adapter.py) acts as a standardized translation layer. Every edge check-in, registration, and incident is formatted into the NHAI Standard Unified Event Format (UEF-1.0), which encapsulates the transaction in a metadata envelope.
Events are written to an offline SQLite queue (datalake_event_queue). A background Service Worker monitors connectivity and automatically flushes the queue via POST requests to the Datalake API when online. For permanently airgapped locations, the system provides an airgapped batch export function to dump pending events to an encrypted JSON file on local drives.

5. SYSTEM WORKFLOW

The FaceShield EdgeAI operational workflow executes in ten sequential, secure steps entirely at the edge gate:

```text
[Camera Feed] ──> 1. User Capture ──> 2. Face Detection ──> 3. Face Alignment 
                      │
                      ├──> 4. Quality Validation ──> [Reject if variance < 1e-4]
                      │
                      ├──> 5. Liveness Scorer (Laplacian / Specular / HSV)
                      │
                      ├──> 6. Embedding Generator (ONNX ArcFace 512D)
                      │
                      ├──> 7. Offline Matching (1:N SQLite Cosine Similarity)
                      │
                      └──> 8. Decision Engine (Identity Trust score + PPE audit)
                                │
                                ├──> 9. Local SQLite Audit Logs (worker_journey)
                                │
                                └──> 10. Result Delivery (WebSocket gateway / Kiosk Gate Open)
```

1. User Capture
The edge gate camera captures a raw, high-resolution BGR frame at 30 FPS. The image is loaded directly into an OpenCV numpy array in local memory, minimizing frame buffer overhead.

2. Face Detection
The BGR frame is converted to RGB, resized to a compact 320x240 canvas, normalized between -1.0 and 1.0, and ingested by the local ONNX UltraFace session. It outputs bounding box coordinates and face confidence scores. An IoU-based Non-Maximum Suppression (NMS) filters overlapping boxes, selecting the primary face with a score > 0.70.

3. Face Alignment
Using the bounding box, the coordinates are mapped back to the original frame resolution. The engine crops the primary face region and dynamically aligns it to standard boundaries, crop constraints, and aspect ratios.

4. Quality Validation
To prevent system bypasses using flat vectors or uniform color masks, the face crop is audited. The face embedding vector variance is calculated. If the variance is less than 1e-4 (flat mock vector check), the transaction is flagged as a mock biometric attack and blocked.

5. Liveness Detection
The aligned face crop is passed to the 3-Signal Passive Liveness Engine. Laplacian texture variance measures edge details (sharp skin vs blurred photo). Specular analysis measures the ratio of glare spots. HSV skin saturation std-dev validates color spread. A composite score is computed:
   Composite = (0.50 * Texture) + (0.25 * Specular) + (0.25 * Skin Saturation)
If the composite is < 0.40, a Spoof Attempt is registered and blocked.

6. Embedding Generation
The live face crop is resized to 112x112, normalized to a range of -127.5 to 128.0, and processed by the ArcFace ONNX model to extract a deep 512-dimensional vector embedding representing facial keypoints. The vector is L2-normalized to ensure unit length.

7. Offline Matching
The 512D unit vector is matched against the local SQLite database (`face_embeddings` table) within the active tenant scope. The system computes the cosine similarity against all cached candidates using in-memory matrix dot products:
   Similarity = DotProduct(QueryVector, CandidateVector)
The match threshold is set to 0.55. If a candidate matches with similarity >= 0.82, it triggers a duplicate identity flag to prevent ghost worker registration.

8. Decision Engine
The composite Identity Trust Score is computed at the edge:
   Trust Score = (FaceMatch * 0.40) + (Liveness * 0.25) + (GPS * 0.15) + (Device * 0.10) + (Behavior * 0.10)
- GPS confidence uses Haversine formula against Site coordinates.
- Device trust scores the Kiosk hardware ID.
- Behavior score analyzes historical check-in hours and recent risk events.
Simultaneously, the OpenCV PPE engine checks for helmets and high-vis vests.
- If Trust Score >= 0.80 and PPE is compliant, the gate is GRANTED.
- If Trust Score is between 0.45 and 0.60, it triggers MANUAL_REVIEW.
- If Trust Score < 0.45 or PPE fails, the gate is DENIED.

9. Audit Logging
The transaction event is recorded locally in the SQLite `worker_journey` and `worker_risk_log` tables. If a violation (geofence, spoof, or watchlist hit) occurs, risk scores are updated. All entries are also written to the local Datalake Queue (`datalake_event_queue`) using the standardized UEF-1.0 schema.

10. Result Delivery
The gate decision is instantly sent to the frontend kiosk interface, rendering a green/red visual response using high-end glassmorphic UI cards. A WebSocket signal pushes the event to supervisors, and a hardware relay trigger opens the physical gate lock.

7. GUIDED ENROLLMENT WORKFLOW

The Guided Enrollment Workflow is a multi-step compliance process that guarantees only high-quality, verified biometric records are entered into the NHAI identity registry, eliminating duplicate identities and "ghost workers" at the source:

```text
  [Initial Login] 
         │
         v
  [1. Must Change Password] ──> [2. Set Secure Credentials]
                                       │
                                       v
  [3. Capture Face] <── [Verify Normal variance >= 1e-4]
         │
         ├──> [4. Laplacian Quality Check] ──> [Reject if blurred]
         │
         ├──> [5. Specular highlight Validation] ──> [Reject if flat photo]
         │
         └──> [6. HSV Skin Tone Distribution Check] ──> [Reject if B&W printout]
                                       │
                                       v
  [7. ArcFace 512D Vector Generation]
         │
         v
  [8. Duplicate Prevention Check (1:N postgres/SQLite)] ──> [Reject if similarity >= 0.82]
         │
         v
  [9. Relational vector Write (Unsupported("vector(512)") in Postgres)]
         │
         v
  [10. AES-256-CBC Fingerprint Encryption & Write]
         │
         v
  [11. Local SQLite Mirror Sync] ──> [Biometric Pending Flag Cleared]
```

* User Onboarding & Registration
When a new contractor or worker is onboarded, their profile is initialized in the relational model with the status `WorkerState.INVITED` or `WorkerState.REGISTERED`. On their initial login, they are flagged with `mustChangePassword: true` and `biometricPending: true` to restrict access until enrollment is fully validated.

* Face Registration & Multi-Angle Capture
The user is guided through a dynamic frontend camera capture wizard. The UI prompts the user to rotate their head slightly (multi-angle capture) to record diverse geometry details and verify a three-dimensional subject.

* Image Quality & Enrollment Validation
The captured frames undergo rigorous edge-side audits:
1. Laplacian Blur Check: Rejects out-of-focus captures.
2. Flat Vector Check: Rejects uniform color bypasses (variance >= 1e-4).
3. 3-Signal Passive Liveness check: Analyzes Laplacian texture, specular highlights, and skin-tone HSV spreads, ensuring the user is a live human.

* Template Generation & Secure Storage
Once quality is validated:
1. The system extracts a 512-dimensional vector embedding using the ONNX ArcFace model.
2. The embedding is matched against the entire existing database using PostgreSQL pgvector. If a match is found with similarity >= 0.82 (duplicate prevention threshold), the system blocks enrollment and logs a `DUPLICATE_IDENTITY` check-in conflict, preventing "ghost worker" registrations.
3. If unique, the embedding is saved in PostgreSQL as an `Unsupported("vector(512)")` column. Simultaneously, fingerprint minutiae keypoint templates are encrypted using AES-256-CBC (key derived via `crypto.scryptSync` from `JWT_SECRET`) and saved.
4. The local SQLite mirror is synchronized, clearing the `biometricPending` flag.

* Table Descriptions
1. `users` (PostgreSQL): Core workforce profiles. Stores metadata, role strings, and `faceEmbedding` as `Unsupported("vector(512)")` columns.
2. `tenants` (PostgreSQL): Multi-tenant context isolators. Holds tenant names, slugs, and organization codes (e.g., `SHIELD`).
3. `sites` (PostgreSQL): Geographic workspaces. Holds lat/long coordinates and geofence radii in meters.
4. `attendance` (PostgreSQL): Gate transaction logs. Tracks check-in/out times, location coordinates, liveness scores, and composite trust scores.
5. `incidents` (PostgreSQL): Security alarms. Records violations (`SPOOF_ATTEMPT`, `GEOFENCE_VIOLATION`) and severity scales (`LOW` to `CRITICAL`).
6. `face_embeddings` (SQLite Local): Active cache of face embeddings. Holds serialised JSON vector strings for local matching.
7. `worker_journey` (SQLite Local): Logs offline worker zone movements, PPE safety compliances, and average trust metrics.

* Relationships
- One Tenant has many Users, Sites, and Attendance records.
- One User belongs to one Tenant, has one UserBiometrics record, and has many Attendance, Incident, and WorkerSite records.
- One Site has many Kiosks, WorkerSites, and belongs to one Tenant.
- User table implements a self-relation (`reportsTo` -> `user_id`) to track subordination matrices.

* Indexing Strategy
To ensure sub-millisecond retrieval times:
1. `idx_user_role`: Speeds up role-based queries.
2. `idx_tenant`: Isolates queries within a single tenant boundary.
3. `idx_tenant_role`: Optimizes multi-tenant command control audits.
4. `idx_state`: Filters active vs suspended workers.
5. pgvector index: Compares high-dimensional embeddings efficiently in PostgreSQL.

* Audit Logging Design
Primary database sizes are kept lean by offloading high-frequency audit logs and raw biometric inference results asynchronously into MongoDB collections (`audit_logs`, `inferences`, `telemetry`). This avoids relational database bloating and locks.

* Face Template Storage Strategy
1. Primary Storage: Unsupported("vector(512)") in PostgreSQL, optimized for cosine distance search queries.
2. Edge Cache: JSON-serialised float arrays (embedding_json) stored in SQLite face_embeddings, optimized for high-speed local numpy scans.
3. Fingerprint storage: Keypoint maps encrypted with AES-256-CBC, stored as base64 string templates.

11. SECURITY ARCHITECTURE

FaceShield EdgeAI establishes an uncompromisable, zero-trust security perimeter combining local cryptographic protections, hardware validation, and a multi-signal offline trust model:

```text
  [Raw Video Frame] ──> Laplacian / Specular highlight / HSV Skin Tone Scans
                             │
                             ├──> [FAIL] ──> Log CRITICAL Spoof Incident & Block
                             v
  [ArcFace Crop] ────> Variance Trap Audit (variance >= 1e-4 check)
                             │
                             ├──> [FAIL] ──> Log Mock Vector Incident & Block
                             v
  [512D Vector] ─────> Duplicate Identity Check (1:N PostgreSQL / SQLite)
                             │
                             ├──> [Similarity >= 0.82] ──> Log Ghost Worker & Block
                             v
  [SQLite Mirror] ───> Multi-Factor Identity Trust Score Compiler
                             │
                             ├──> GPS Coordinates -> Haversine Geofence check
                             ├──> Device Kiosk ID -> Known hardware register
                             └──> User History -> Time-of-day pattern audit
                                        │
                                        v
                            [Composite Score Compiled]
                                        │
                                        ├──> Trust >= 0.80 ──> Access GRANTED
                                        ├──> Trust >= 0.45 ──> MANUAL_REVIEW
                                        └──> Trust <  0.45 ──> Access DENIED
```

* Encryption
All sensitive biological credentials (fingerprints) are encrypted in PostgreSQL using AES-256-CBC. Key derivation is performed dynamically via `crypto.scryptSync` from the secure environment `JWT_SECRET`. Local SQLite files are stored on encrypted operating system partitions (BitLocker / LUKS) to prevent physical storage attacks.

* Face Template Protection
Biometric embeddings are deep neural features (ArcFace 512D vectors) which cannot be reverse-engineered to reconstruct the original face image, satisfying state and citizen privacy mandates.

* Authentication & Authorization
Granular 9-Tier Role-Based Access Control (RBAC) is enforced at the API level via NestJS guards. Platform-level endpoints (tenant provisioning) are isolated to `PLATFORM_HEAD` roles, bypassing standard tenant boundaries while logging platform audits.

* Secure Storage
Local browser IndexedDB records are encrypted using AES, preventing local client profile tampering. Primary transactional relations reside on secure isolated database networks.

* Anti-Spoofing
The passive liveness engine combines texture, reflection, and hue spreads to block static photos and screen replays, while the variance trap prevents flat vector injection attacks.

* Audit Trails
Every security breach (spoof attempt, geofence breach, or watchlist match) immediately triggers an `Incident` log. Spoofs are logged at `CRITICAL` severity, geofences at `HIGH` severity. Telemetry logs monitor edge CPU/RAM usage to audit terminal health.

* Device Security
Kiosk systems are registered in the database with unique hardware IDs (`deviceId`). The Trust Score compiler checks Kiosk registers, assigning a lower Device Trust score to queries coming from unregistered browsers.

* Offline Trust Model
Rather than a single weak link, the system relies on a composite model where face similarity, liveness, GPS boundaries, hardware registry, and historical timesheet deviation are weighted. A low score on one factor (e.g., GPS drift) doesn't completely block access but raises an advisory, while liveness failures block entry immediately.

FaceShield EdgeAI represents a major shift from traditional biometric systems, introducing four primary technical differentiators:

* Unique AI Approaches

Matrix calculations are handled using NumPy for maximum performance.

* Lightweight Architecture
Decoupled Dual-Database mirrors allow massive 1:N searches entirely on the edge. PostgreSQL handles cloud indexing while SQLite buffers edge check-ins. High-frequency telemetry and audit logs are offloaded to MongoDB, keeping transactional databases lean and fast.

* Novel Workflows
The system integrates biometric checks with mandatory OpenCV-based safety gear auditing (helmet, vest, mask). Access is granted only when identity is confirmed and safety gear is worn, reducing site liabilities.

* Privacy-Preserving Design
All face templates are stored as normalized mathematical vectors that protect user identities. Fingerprints are encrypted using AES-256-CBC, preventing keypoint theft.

* Government-Scale Readiness
The system uses the NHAI Unified Event Format (UEF-1.0) and SQLite event queuing. Kiosks operate in complete isolation, syncing data asynchronously when networks become available, and supporting manual drive exports in extreme airgapped conditions.
